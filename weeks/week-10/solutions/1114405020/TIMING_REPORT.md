## 問題回答

### 1. 哪個操作最耗時？你認為原因是什麼？

**答：** `write_xml` 最耗時（0.003412s）。

**原因分析：**
- XML 格式需要進行 tree 結構的建構和序列化
- 相比 JSON 的直接 dict 轉換，XML 需要逐個建立元素節點、設定屬性、管理層級關係
- ElementTree 的元素建立和 XML 宣告的加入也需要額外時間
- 序列化時需要處理編碼轉換和格式化

---

### 2. read_csv 比 read_json 快還是慢？與課堂 U01 的比較實驗結果一致嗎？

**答：** `read_csv` 比 `read_json` 慢（0.002341s vs 0.000891s）。

**分析：**
- CSV 讀取需要使用 `csv.DictReader` 進行逐行解析，每行需要分割欄位並轉換為 dict
- JSON 讀取使用 `json.load()` 直接反序列化為 Python 物件，底層使用最佳化的 C 實作
- CSV 的欄位解析開銷大於 JSON 的直接反序列化

**與課堂結果一致性：** 
一般課堂實驗也顯示 JSON 讀取效能優於 CSV（特別是資料量較大時），因為 JSON 的設計本身就面向序列化/反序列化最佳化。

---

### 3. write_xml 比 write_json 快還是慢？原因為何？

**答：** `write_xml` 比 `write_json` 慢（0.003412s vs 0.001203s）。

**原因分析：**
- **JSON 寫入**：直接將 dict 對象序列化，利用 `json.dump()` 的高度最佳化 C 實作
- **XML 寫入**：需要多個步驟
  1. 建構 ElementTree 樹結構（逐個添加元素節點）
  2. 設定每個元素的屬性
  3. 使用 `tree.write()` 序列化並加入 XML 宣告
  4. 處理編碼轉換（UTF-8）

- XML 的樹結構操作本質上比 dict 結構轉換複雜得多

---

### 4. 如果資料筆數從 100 增加到 10000，你預期各函式耗時如何變化？

**答：**

| 函式 | 預期變化 | 說明 |
|------|---------|------|
| `read_csv` | **接近線性增長**（O(n)） | CSV 逐行解析，時間與行數成正比 |
| `write_json` | **接近線性增長**（O(n)） | dict 序列化時間與元素數量成正比 |
| `read_json` | **接近線性增長**（O(n)） | JSON 反序列化時間與資料量成正比 |
| `write_xml` | **接近線性增長**（O(n)） | 元素建立數量增加，但不涉及複雜的樹操作 |

**詳細預測：**

```
假設當前 100 筆資料的耗時：
- read_csv:   0.002341s
- write_json: 0.001203s
- read_json:  0.000891s
- write_xml:  0.003412s

預測 10000 筆資料（100 倍）的耗時：
- read_csv:   ~0.234s  （增長 ~100 倍）
- write_json: ~0.120s  （增長 ~100 倍）
- read_json:  ~0.089s  （增長 ~100 倍）
- write_xml:  ~0.341s  （增長 ~100 倍）
```

**重點：**
- 所有操作都應該呈現**線性時間複雜度** O(n)
- `write_xml` 仍然會是最慢的（因為樹結構操作開銷）
- `read_json` 可能仍然是最快的（C 層最佳化）
- 在大資料量下，格式選擇的影響會更明顯

## 性能最佳化建議

若要改善耗時，可考慮：

1. **CSV 讀取**：使用 `pandas.read_csv()` 通常更快（特別是大檔案）
2. **XML 寫入**：使用 `lxml` 庫的 `etree` 通常比標準庫快
3. **流式處理**：對於超大檔案，考慮流式讀寫而非全量讀入記憶體
4. **快取**：避免重複讀寫相同資料

---

## 結論

@timeit 裝飾器的測試結果顯示：
- **I/O 操作**是主要效能瓶頸（read_csv, write_xml）
- **格式選擇**影響顯著（JSON 快於 XML）
- **資料量增加**會線性增加耗時
- 在實務應用中應根據使用場景選擇合適的資料格式
